NitroKV Command Specification

## 1. Общие правила

### 1.1. Формат клиентской команды

Каждая клиентская команда должна передаваться в виде:

- `RespArray`
- каждый элемент массива — `RespBulkString`

Иными словами, команда должна иметь вид:

```text
RespArray[RespBulkString, RespBulkString, ...]
Первый элемент массива — имя команды.
Остальные элементы — аргументы команды.
```

### 1.2. Arity

Для каждой команды задаётся **arity** — допустимое количество аргументов **без учёта имени команды**.

Arity описывается двумя значениями:

*   `min` — минимальное число аргументов

*   `max` — максимальное число аргументов


Пример:

*   `GET key` → `arity = {1, 1}`

*   `EXPIRE key seconds [NX|XX|GT|LT]` → `arity = {2, 3}`


* * *

### 1.3. Общие замечания

*   Имена ключей передаются как `RespBulkString`

*   Значения передаются как `RespBulkString`

*   Числовые аргументы команд также передаются как `RespBulkString` и затем интерпретируются parser'ом команды

*   Ошибки в типах аргументов, количестве аргументов или формате аргументов должны приводить к ошибке парсинга команды

3\. Command Reference
---------------------

* * *

3.1. `SET`
----------

### Syntax

SET key value

### Arity

2..2

### Arguments

| Argument | Type | Description |
| --- | --- | --- |
| `key` | `RespBulkString` | Имя переменной / ключ |
| `value` | `RespBulkString` | Значение переменной |

### Semantics

*   Устанавливает значение переменной

*   Если переменная уже существует, её значение обновляется


* * *

3.2. `GET`
----------

### Syntax

GET key

### Arity

1..1

### Arguments

| Argument | Type | Description |
| --- | --- | --- |
| `key` | `RespBulkString` | Имя переменной / ключ |

### Semantics

*   Возвращает значение переменной, если она существует

*   Если переменная не существует, возвращается ошибка или специальный ответ, определённый логикой сервера


&gt; Точное поведение для отсутствующего ключа должно быть зафиксировано отдельно в reply semantics.

* * *

3.3. `DEL`
----------

### Syntax

DEL key

### Arity

1..1

### Arguments

| Argument | Type | Description |
| --- | --- | --- |
| `key` | `RespBulkString` | Имя переменной / ключ |

### Semantics

*   Удаляет переменную, если она существует

*   Если переменная не существует, возвращается ошибка или специальный ответ, определённый логикой сервера


&gt; Поведение лучше зафиксировать отдельно: ошибка это или integer result (`0/1`).

* * *

3.4. `PING`
-----------

### Syntax

PING

### Arity

0..0

### Semantics

*   Не принимает аргументов

*   Возвращает `Simple String` со значением `PONG`


### Expected reply

+PONG

* * *

3.5. `EXPIRE`
-------------

### Syntax

EXPIRE key seconds \[NX|XX|GT|LT\]

### Arity

2..3

### Arguments

| Argument | Type | Description |
| --- | --- | --- |
| `key` | `RespBulkString` | Имя переменной / ключ |
| `seconds` | `RespBulkString` | Положительное целое число |
| \`\[NX | XX | GT |

### Semantics

*   Устанавливает TTL для переменной, если она существует

*   Если TTL уже установлен, обновляет его

*   Если переменная не существует, возвращается ошибка


### Optional modifiers

| Modifier | Meaning |
| --- | --- |
| `NX` | Установить TTL только если у ключа ещё нет TTL |
| `XX` | Установить TTL только если у ключа уже есть TTL |
| `GT` | Установить TTL только если новый TTL больше текущего |
| `LT` | Установить TTL только если новый TTL меньше текущего |

### Current implementation note

На текущем этапе модификаторы:

*   распознаются синтаксически

*   но могут игнорироваться логикой обработки


Это поведение должно быть либо явно задокументировано, либо доведено до полной реализации.

* * *

3.6. `TTL`
----------

### Syntax

TTL key

### Arity

1..1

### Arguments

| Argument | Type | Description |
| --- | --- | --- |
| `key` | `RespBulkString` | Имя переменной / ключ |

### Semantics

*   Возвращает оставшееся TTL для переменной

*   Если переменная не существует, возвращается ошибка или специальный результат, определённый логикой сервера


&gt; Здесь тоже желательно отдельно зафиксировать reply semantics:  
&gt; например, ошибка / `-1` / `-2` / integer remaining lifetime.

* * *

3.7. `EXISTS`
-------------

### Syntax

EXISTS key

### Arity

1..1

### Arguments

| Argument | Type | Description |
| --- | --- | --- |
| `key` | `RespBulkString` | Имя переменной / ключ |

### Semantics

Возвращает целочисленный результат:

*   `1` — если переменная существует

*   `0` — если переменная не существует


* * *

4\. Parser Notes
----------------

Command parser должен выполнять следующие проверки:

1.  входное значение должно быть `RespArray`

2.  массив не должен быть пустым

3.  имя команды должно быть `RespBulkString`

4.  количество аргументов должно соответствовать `arity`

5.  типы аргументов должны соответствовать спецификации команды

6.  строковые числовые аргументы должны корректно преобразовываться в числа

7.  недопустимые опции должны приводить к ошибке парсинга


* * *

5\. Open Questions
------------------

Следующие моменты должны быть зафиксированы отдельно:

*   reply semantics для `GET`, если ключ отсутствует

*   reply semantics для `DEL`, если ключ отсутствует

*   reply semantics для `TTL`, если ключ отсутствует или TTL не установлен

*   считать ли отсутствие ключа ошибкой или штатным результатом

*   поддерживаются ли модификаторы `EXPIRE` только синтаксически или уже логически”
